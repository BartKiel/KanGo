💾 Backup Policy — app.stratosx.com

Comprehensive backup strategy for the StratosX GEO Analytics Platform deployed on Google Cloud Run + GCS.

1. 📋 Overview

Architecture Summary

  • Application: StratosX (server.py, ~7026 lines, Python/Flask)
  • Hosting: Google Cloud Run (service: stratosx)
  • Data Store: Google Cloud Storage (GCS) — all data stored as JSON/XLSX files
  • GCP Project: allegro-autopilot-app
  • Region: europe-central2 (Warsaw, Poland)
  • Source Code: GitHub BartKiel/stratosx-deploy (private)
Key Principle: StratosX is stateless. All persistent data lives in GCS buckets. Cloud Run containers are ephemeral and can be redeployed at any time from GitHub source. This makes backups focused on GCS data and Git history.

3. ⏰ Backup Schedule

Daily Automated Backup Automated

  • Trigger: Google Cloud Scheduler job stratosx-daily-backup
  • Schedule: Daily at 03:00 CET (01:00 UTC)
  • Method: HTTP POST to /cron/daily-backup with CRON_SECRET header
  • Target Bucket: stratosx-backups-daily
  • Path Format: YYYY-MM-DD/<filename>
  • Scope: Full copy of all blobs from production bucket

Git Pushes Manual

  • Every code change is committed and pushed to GitHub
  • Cloud Run stores revision history (last 100 revisions)
  • Rollback: set traffic to any previous revision via GCP Console

5. 🔄 Restore Procedures

Scenario A: Restore All Data (Full Recovery)

  1. Identify the backup date: gsutil ls gs://stratosx-backups-daily/
  2. Copy all files back to production:
    gsutil -m cp -r gs://stratosx-backups-daily/2026-03-10/* gs://stratosx-app-data/
  3. Verify app is working: visit app.stratosx.com

Scenario B: Restore Single File

  1. Find the file in backup: gsutil ls gs://stratosx-backups-daily/2026-03-10/users.json
  2. Copy it back: gsutil cp gs://stratosx-backups-daily/2026-03-10/users.json gs://stratosx-app-data/users.json

Scenario C: Rollback Application Code

  1. Go to GCP Console → Cloud Run → stratosx → Revisions
  2. Find the previous working revision
  3. Click "Manage Traffic" → Route 100% to that revision
  4. OR: git revert + gcloud run deploy
6. 🔒 Security
  • Backup auth: CRON_SECRET environment variable — Cloud Scheduler sends it in header, endpoint validates
  • GCS access: Cloud Run service account has storage.objectAdmin on relevant buckets
  • Encryption: GCS default encryption (Google-managed keys, AES-256)
  • GitHub: Private repositories, 2FA enabled
  • Passwords: SHA-256 hashed in users.json, no plaintext
7. 📊 Monitoring & Alerts
  • Cloud Scheduler: Check execution logs in GCP Console → Cloud Scheduler → Logs
  • Backup verification: gsutil ls gs://stratosx-backups-daily/ | tail -5 — confirm latest date exists
  • Manual health check: visit app.stratosx.com, login, verify dashboards load
✅ Quick Backup Check Command:
gsutil du -sh gs://stratosx-backups-daily/$(date +%Y-%m-%d)/
