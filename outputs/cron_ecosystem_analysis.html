Opcja A: Cloud Scheduler → bezpośrednio Cloud Run (REKOMENDOWANA)

Najprostsza opcja. Cloud Scheduler wywołuje HTTP endpoint na istniejącym serwisie Cloud Run. Nie potrzebujemy nowej infrastruktury — wystarczy dodać endpoint /cron/<task_name> do istniejącej Flask app i skonfigurować scheduler.

# Krok 1: Dodaj endpoint do app.py (np. KanGo)
@app.route("/cron/daily-backup", methods=["POST"])
def cron_daily_backup():
    token = request.headers.get("X-Cron-Token")
    if token != os.environ.get("CRON_SECRET"):
        return "Unauthorized", 403
    # --- tutaj logika cron'a ---
    do_backup()
    return "OK", 200

# Krok 2: Utwórz Cloud Scheduler job (jednorazowo)
# gcloud scheduler jobs create http daily-backup \
#   --schedule="0 3 * * *" \
#   --uri="https://kango-969597502498.europe-west1.run.app/cron/daily-backup" \
#   --http-method=POST \
#   --headers="X-Cron-Token=SECRET_VALUE" \
#   --location=europe-west1 \
#   --time-zone="Europe/Warsaw"

Zalety: zero nowej infrastruktury, reuse istniejących serwisów, łatwe testowanie (curl endpoint), monitoring w Cloud Run logs.

Wady: endpoint musi odpowiedzieć w <30 min (limit Cloud Run), serwis musi być aktywny.

Opcja B: Cloud Scheduler → Cloud Function

Cloud Function to samodzielny kawałek kodu Python, który uruchamia się na żądanie. Przydatne gdy logika cron'a jest niezwiązana z żadnym serwisem lub gdy chcemy izolację.

# functions/daily_report/main.py
import functions_framework
from google.cloud import storage

@functions_framework.http
def daily_report(request):
    client = storage.Client()
    bucket = client.bucket("stratosx-data-esj")
    # ... logika raportu ...
    return "Report generated", 200

# Deploy:
# gcloud functions deploy daily-report \
#   --gen2 --runtime=python312 --region=europe-west1 \
#   --source=. --entry-point=daily_report \
#   --trigger-http --no-allow-unauthenticated

# Scheduler:
# gcloud scheduler jobs create http daily-report \
#   --schedule="0 8 * * 1" \
#   --uri="FUNCTION_URL" \
#   --oidc-service-account-email="SA@PROJECT.iam.gserviceaccount.com"

Zalety: izolacja, nie obciąża Cloud Run, auto-scaling, do 60 min timeout.

Wady: nowy typ zasobu do zarządzania, cold starts (~2-5s), osobny deploy.

Opcja C: Cloud Scheduler → Pub/Sub → Cloud Run

Najbardziej rozbudowana opcja. Przydatna przy łańcuchach zadań (task A → task B → task C) lub gdy potrzebujemy retry logic i dead-letter queue. Overkill na nasze obecne potrzeby.

Wady: Pub/Sub topics, subscriptions, IAM permissions, push endpoints — dużo moving parts.

3 Krok po kroku — jak to wdrożyć (Opcja A)

1. Włącz Cloud Scheduler API

gcloud services enable cloudscheduler.googleapis.com — jednorazowo, 10 sekund.

2. Dodaj endpoint /cron/* do Flask app

Prosty endpoint z autoryzacją tokenem. ~15 linijek kodu. Testuj lokalnie z curl -X POST.

3. Dodaj CRON_SECRET do Secret Manager

gcloud secrets create cron-secret + update Cloud Run env var. ~2 min.

4. Deploy Cloud Run z nowym endpointem

Standardowy gcloud run deploy — jak zawsze. ~3 min.

5. Utwórz Cloud Scheduler job

Jedna komenda gcloud scheduler jobs create http. ~1 min. Gotowe!

🎯 Werdykt: NIE, to nie jest skomplikowane

Wdrożenie cron'a w naszym ekosystemie jest zaskakująco proste. Opcja A (Cloud Scheduler → Cloud Run endpoint) wymaga:

  • ~15 linii kodu w Flask app
  • 1 komenda gcloud do utworzenia schedulera
  • 1 sekret w Secret Manager
  • 15-30 minut pracy na pierwszy cron job
  • $0.00 / miesiąc kosztów

Każdy kolejny cron job to ~5 minut (dodaj endpoint + scheduler job). Nie potrzebujemy żadnej nowej infrastruktury — reuse'ujemy istniejące Cloud Run services.
