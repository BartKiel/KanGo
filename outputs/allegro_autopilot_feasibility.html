🎯 What Is Allegro Autopilot?

An automated Featured bidding & inventory management system for Allegro sellers. It abstracts bid/budget management into a single "Featured Intensity" control, provides guardrails to protect against overspend, and classifies SKUs to optimize allocation.

Primary KPI

Units sold (absolute increase)

Secondary KPIs

ACoS / ROAS, conversion rate, organic uplift

Architecture

Multi-tenant SaaS, one Allegro account per tenant

🧩 Module Breakdown (MVP)

1. Featured Intensity Engine

CRITICAL PATH

Single "intensity dial" abstraction over Allegro's Featured API. Manages bids/budgets automatically with daily cadence (configurable).

Effort: 4–5 weeks
Complexity: High
Dependency: Allegro API access

2. Guardrails & Anomaly Detection

CRITICAL PATH

Hard max daily spend, stock-level pause, automatic taper, global kill switch, spend spike / conversion drop detection.

Effort: 3–4 weeks
Complexity: Medium-High

3. SKU Classification Engine (Rules-Based)

IMPORTANT

Classify SKUs as Growth, Profitable-Steady, Underperformer, Dead-Weight, or Seasonal. Rules-based for MVP (AI-based in Phase 2).

Complexity: Medium

🏔️ Key Technical Challenges

1. Allegro Featured API Integration

The Featured (Oferty Promowane) API requires partner-level access. Documentation is partly in Polish, and bid mechanics change. We need a sandbox environment and real account for testing.

2. Safe Bidding Without Burning Budget

Automating spend is inherently risky. The system needs robust circuit breakers: hard spend caps, anomaly detection (Z-score on hourly spend), and graceful degradation to manual mode.

3. Multi-Tenant Data Isolation

Each tenant manages real money. Cross-tenant data leakage is unacceptable. Need proper tenant-scoped DB schemas, API-key isolation, and audit logging from day 1.

4. Organic Lift Measurement

Attributing sales to "organic lift from Featured" is inherently noisy. Need clean baseline measurement, controlled taper experiments, and statistical significance checks.
