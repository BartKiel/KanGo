🎯 1. Czym jest KanGo

KanGo to lekka aplikacja Kanban zbudowana specjalnie dla ekosystemu StratosX. Służy do zarządzania zadaniami — od rekomendacji AI, przez production, po testy i delivery.

Copilot pracuje z KanGo jako swój główny system zadań: pobiera taski z bucketu for-production, realizuje je, dodaje komentarze z wynikami, i przenosi do for-tests-confirm.

🔄 5. Workflow pracy Copilota

Gdy użytkownik mówi "znow for production" lub "go for another for production":

  1. Pobierz taski z for-production (GET /api/tasks, filtruj po bucket)
  2. Dla każdego taska:
    • Ustaw in_progress: true (PUT /api/tasks/<id>)
    • Zrealizuj task (napisz kod, uruchom komendy, stwórz pliki)
    • Dodaj comment z wynikami (zaczynaj od ✅)
    • Ustaw done: true, in_progress: false, bucket: "for-tests-confirm"
  3. Jeśli task wymaga deploy — deploy po zmianach
  4. Jeśli task tworzy plik analityczny — zapisz w KanGo/outputs/ + upload do /api/files

⚠️ Ważne zasady

  • Nie ustawiaj in_progress na wszystkich for-production naraz — pracuj po jednym lub w małych grupach
  • Board ma live polling (5s) — użytkownik widzi na żywo, nad czym Copilot pracuje
  • in_progress: true → karta pulsuje niebieskim glow na boardzie
  • Po przeniesieniu do for-tests-confirm → karta ma zielony glow

📁 6. System plików

Dual-save rule (RUNBOOK §15)

Każdy plik analityczny/raportowy musi być zapisany w dwóch miejscach:

  1. Lokalnie: KanGo/outputs/nazwa_pliku.html
  2. W GCS: upload przez POST /api/files (multipart)

Co uploadować

  • ✅ HTML raporty, prezentacje, analizy
  • ✅ Excel/CSV dane
  • ✅ PDF, Markdown
  • ❌ Pliki źródłowe (.py) — te zostają w repo
  • ❌ Pliki konfiguracyjne (yaml, json, toml)

Upload przez API

curl -b /tmp/kc -X POST https://kango-...run.app/api/files \
  -F "file=@/path/to/file.html"
