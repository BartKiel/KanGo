🎯 1. Problem

GEO-Catch 2 generuje clustery embeddingowe — grupy semantycznie powiązanych fraz/entitetów z wyników AI (np. ChatGPT, Perplexity). Ale nie wszystkie clustery mają taką samą wartość biznesową:

  • Cluster „karma dla psów premium" może mieć 15 000 MSV i CPC $2.50
  • Cluster „karma dla psów z grzybami reishi" może mieć 40 MSV i CPC $0.10

Bez ważenia oba clustery mają taką samą „wielkość" na mapie embeddingowej — co jest mylące. Potrzebujemy systemu wag opartego o realne dane rynkowe.

🧮 4. Formuła ważenia — Cluster Weight Score

Proponuję Cluster Weight Score (CWS) — złożoną wagę łączącą wolumen, wartość i szansę:

CWS = α · V_norm + β · $V_norm + γ · Opp_norm

gdzie α + β + γ = 1.0

Składniki

SymbolMetrykaFormuła per cluster
VVolume Scorelog₂(1 + SUM(msv))
$VValue Scorelog₂(1 + SUM(msv × cpc))
OppOpportunity ScoreSUM(msv × (1 - competition) × (1 - kd/100))

💡 Logarytm (log₂) jest kluczowy — bez niego dominujące clustery (np. „karma dla psów" z MSV 100k) miałyby wagę 1000x większą niż long-tail, co czyniłoby mapę nieczytelną.

📐 6. Normalizacja i skalowanie

Problem rozkładu

Dane keyword mają power-law distribution — kilka head terms ma MSV 100k+, większość long-tail ma MSV < 500. Bez transformacji mapa byłaby zdominowana przez 2-3 clustery.

Rozwiązanie: Log + Min-Max

import numpy as np

def normalize_cws(clusters):
    # 1. Log-transform surowych wartości
    for c in clusters:
        c['v_log']   = np.log2(1 + c['total_msv'])
        c['val_log'] = np.log2(1 + c['total_value'])
    
    # 2. Min-Max normalizacja do [0, 1]
    for key in ['v_log', 'val_log', 'opportunity']:
        vals = [c[key] for c in clusters]
        mn, mx = min(vals), max(vals)
        rng = mx - mn if mx != mn else 1
        for c in clusters:
            c[f'{key}_norm'] = (c[key] - mn) / rng
    
    # 3. CWS z wagami profilu
    alpha, beta, gamma = 0.40, 0.35, 0.25
    for c in clusters:
        c['cws'] = (alpha * c['v_log_norm'] + 
                    beta  * c['val_log_norm'] + 
                    gamma * c['opportunity_norm'])
    return clusters

🗺️ 7. Wizualizacja na mapie

Wariant A: Bubble chart na 2D UMAP

Jak teraz, ale z wagami:

  • Pozycja (x, y) — z UMAP redukcji embeddingów centroidu clustera
  • Rozmiar bąbla — CWS (zamiast count elementów)
  • Hover tooltip: nazwa clustera, CWS, MSV, CPC, keywords count

Wariant B: Treemap (alternatywa)

Płaski treemap gdzie:

  • Rozmiar prostokąta = CWS
  • Kolor = opportunity score (zielony = duża szansa)
  • Łatwiejszy do czytania dla klientów (nie wymaga rozumienia embeddingów)

Wariant C: Hybrid (rekomendacja)

Mapa UMAP + sidebar z treemap + filtr profilów (Balanced/Traffic/Revenue/Opportunity). Użytkownik przełącza profil wagowy i widzi jak zmienia się mapa — clustery „nabierają" lub „tracą" rozmiar.

🚀 9. Kolejne kroki (implementacja)

Ta propozycja to myślenie — poniżej roadmap implementacji:

1
DataForSEO Integration — Endpoint do pobierania MSV/CPC per lista fraz. Batch API call (max 700 keywords/request). Estymacja: 2h.
2
Cluster Aggregator — Moduł w GEO-Catch 2 backend: po clusteringu → auto-enrich z DataForSEO. Oblicz V, $V, Opp per cluster. Estymacja: 3h.
3
CWS Calculator — Normalizacja + formuła CWS z wyborem profilu (dropdown). Cache wyników (keywords data rzadko się zmienia). Estymacja: 1h.
4
Frontend — Weighted Bubbles — Modyfikacja Plotly/D3 chart: rozmiar bąbla = CWS, tooltip z metrykami. Profile switcher. Estymacja: 3h.
5
Testing — Unit tests dla normalizacji (edge cases z zerami, single-kw clusters). Visual QA na 3+ datasetach. Estymacja: 2h.

📌 Łączna estymacja: ~11h dev time. Większość to integracja DataForSEO + frontend.

📌 DataForSEO API cost: ~$0.01 per 100 keywords → nawet przy 5000 fraz = $0.50/run.
