AI / Automation Architecture Remote Execution

Odpalanie Copilota przez WWW

Czy można uruchamiać zadania AI coding assistanta z poziomu przeglądarki — bez siedzenia przy komputerze? Analiza architektur, narzędzi i konfiguracji.

✅ VERDICT: TAK — jest to możliwe na kilka sposobów

🥇
Najłatwiej: Claude Code + tmux
SSH do maszyny z claude, odpal z telefona via web terminal. Działa TERAZ.
🥈
Najlepiej: GitHub Copilot Agent Mode + Codespaces
VS Code w przeglądarce + Copilot agent. Zero lokalnej konfiguracji.
🥉
Najbardziej custom: KanGo → API → Codex/Claude
Webhook z KanGo triggeruje AI agent na serwerze. Wymaga budowania.

📋 1. Definicja problemu

Aktualny workflow: siedzisz przy Macu → otwierasz VS Code → Copilot/Claude działa lokalnie. Problem: chcesz z poziomu KanGo w przeglądarce (np. z telefonu) zlecić zadanie coding, a pilot ma je wykonać — bez otwierania laptopa.

Wymagania:

  • ✅ Trigger z przeglądarki (w tym mobile)
  • ✅ AI coding agent wykonuje zadanie (edycja kodu, deploy)
  • ✅ Wynik widoczny bez otwierania IDE
  • ✅ Bezpieczeństwo (nie każdy może triggerować)
  • ⭐ Bonus: integracja z KanGo board (task → auto-execute)

🏗️ 2. Dostępne opcje (marzec 2026)

Opcja A: GitHub Codespaces + Copilot Agent

Gotowe Cloud IDE Copilot
Trudność
⭐ Łatwa

GitHub Codespaces to VS Code w przeglądarce. Copilot Agent Mode (od Q1 2025) pozwala zlecić multi-file zadania w trybie agentic. Otwierasz github.dev lub Codespace z telefonu → piszesz prompt → Copilot robi.

Konfiguracja:

  1. Repo KanGo na GitHub ✅ (już jest — BartKiel/KanGo)
  2. GitHub Copilot license (Business lub Pro+) ✅
  3. Otwórz github.com/codespaces → New → KanGo
  4. W VS Code web → Copilot Chat → Agent mode → prompt
  5. Copilot edytuje, commituje, otwiera PR

ARCHITEKTURA

📱 Telefon / Browser
↓
☁️ GitHub Codespaces (cloud VM)
↓
🤖 Copilot Agent Mode (w VS Code web)
↓
📦 Git commit → Cloud Run deploy

✅ Zalety

  • • Działa OOB, zero custom code
  • • Pełne VS Code w przeglądarce
  • • Copilot widzi cały kontekst repo
  • • Działa z telefonu
  • • Bezpieczne (GitHub auth)

❌ Wady

  • • Nadal musisz pisać prompt ręcznie
  • • Nie jest "fire and forget" z KanGo
  • • Codespace startup = ~60s
  • • Koszt: $0.18/h (2-core) → ~$13/mies

Opcja B: Claude Code + SSH + Web Terminal

Gotowe CLI Claude
Trudność
⭐ Łatwa

claude CLI (Claude Code) działa w terminalu bez GUI. Wystarczy SSH do maszyny (Mac/serwer) z dowolnego web-terminala (Termius, Blink Shell, nawet KanGo z wbudowanym xterm.js).

Konfiguracja:

  1. Mac z zainstalowanym claude CLI (masz!)
  2. SSH server na Macu: System Settings → Remote Login ✅
  3. tmux na Macu (dla persystencji sesji)
  4. Web SSH client: Termius (app), lub własny via KanGo
  5. ssh mac → tmux attach → claude "fix bug X in app.py"

ARCHITEKTURA

📱 Telefon → Web SSH (Termius / xterm.js)
↓
💻 Mac (tmux session, claude CLI)
↓
🤖 Claude Code edytuje pliki
↓
📦 git push → Cloud Build → Deploy

💡 PRO TIP: One-liner z telefona

ssh mac "cd ~/KanGo && claude --dangerously-skip-permissions 'Implement the task: fix login timeout bug. Then git commit and push.'"

Claude Code przyjmuje prompt jako argument — zero interakcji, fire-and-forget. Wynik na GitHubie.

Opcja C: GitHub Copilot Workspace (Spark)

Preview GitHub Native
Trudność
⭐⭐ Średnia

GitHub Copilot Workspace (dawniej "Spark") pozwala na uruchomienie Issue → Plan → Implement → PR w pełni z poziomu github.com — bez IDE. Otwierasz issue, klikasz "Open in Copilot Workspace", Copilot generuje plan zmian, implementuje, otwiera PR. Dostępny w Technical Preview (od Q4 2025).

✅ Idealnie do

  • • "Zrób to co w tym issue" z telefona
  • • Zero konfiguracji, działa na github.com
  • • AI czyta issue, tworzy plan, implementuje

❌ Ograniczenia

  • • Nadal w preview (waitlist)
  • • Mniej kontekstu niż Copilot w IDE
  • • Nie deployuje automatycznie

Opcja E: Custom — KanGo → Webhook → Claude API + Git

Custom Build Full Control
Trudność
⭐⭐⭐ Trudna

Zbudować własny pipeline: KanGo task button → Cloud Function → Claude API (z kodem repo w context) → commit wyników → deploy. Pełna kontrola, ale dużo roboty (~2-3 dni dev).

Komponenty do zbudowania:

  1. KanGo UI: Przycisk "🤖 Execute with AI" na task card
  2. API endpoint: POST /api/tasks/:id/execute
  3. Cloud Function: Klonuje repo, ładuje pliki do kontekstu Claude API
  4. Claude API call: System prompt + task prompt + code files → response z edytowanymi plikami
  5. Git ops: Apply diffs, commit, push, trigger Cloud Build
  6. Callback: Update task status w KanGo → done + link do PR
⚠️ Ograniczenie: Claude API nie ma dostępu do file system. Musisz ręcznie ładować kod do kontekstu i parsować output. Anthropic Claude "computer use" / "tool use" to inny flow (desktop agent, nie serverless).

🏆 4. Rekomendowany setup (2 fazy)

Faza 1: Teraz (0 dni pracy)

Claude Code + SSH + tmux. Działa z istniejącą infrastrukturą.

Setup (jednorazowy, ~10 min):

  1. Włącz SSH na Macu: System Settings → General → Sharing → Remote Login
  2. Zainstaluj tmux: brew install tmux
  3. Skonfiguruj tailscale lub ngrok dla dostępu spoza LAN
  4. Na telefonie zainstaluj Termius (iOS/Android) — darmowy SSH client

Daily workflow:

# Z telefona (Termius):

$ ssh bartosz@mac.tailnet

$ cd ~/Documents/Coding_space-Python/KanGo

$ tmux new -s copilot

$ claude "Read the task from KanGo API (task id X). Implement it. Git commit and push. Then deploy with gcloud."

# Detach: Ctrl+B, D — claude pracuje dalej

# Sprawdź potem: ssh → tmux attach -t copilot

🔒 6. Bezpieczeństwo

⚠️ Ryzyka

  • • AI może wygenerować złośliwy kod
  • • Niekontrolowany dostęp do repo
  • • API key leaks
  • • Runaway costs (token loop)

✅ Mitygacja

  • • AI commituje do brancha, nie main
  • • PR review przed merge
  • • Max token budget per execution
  • • Whitelist allowed tasks/repos

🛡️ Must-have

  • • Auth required (KanGo login)
  • • Rate limit: max 10 exec/day
  • • Audit log w GCS
  • • Sandbox: Cloud Run Job (isolated)
